1. Краткий смысл проекта

First AI — это не одна нейросеть и не отдельный «обученный бот». Это переиспользуемая система, состоящая из визуального React-виджета и отдельного backend-сервиса. Виджет можно поставить на разные React-сайты, передать ему описание конкретного проекта, а backend превратит эти данные в контекст для AI-модели.

Один и тот же компонент может быть:

  • консультантом по услугам и ценам;
  • помощником по навигации сайта;
  • первой линией поддержки;
  • помощником по выбору услуги;
  • сборщиком требований и контактных данных внутри диалога;
  • демонстрацией AI-интеграции для клиента.

Поведение меняется не переписыванием внутреннего кода чата, а входными данными: project, visitorName, pageContext, locale, настройками интерфейса и адресом backend.

Главная идея: React-компонент отвечает за интерфейс и отправку данных, Worker отвечает за проверку, безопасное хранение ключей и общение с AI, а сайт-владелец отвечает за реальные бизнес-действия: базу данных, email, Telegram, CRM, согласие пользователя и обработку персональных данных.

2. Цель и границы версии 0.1

2.1. Что уже работает

  • Плавающий адаптивный чат для React 18 и новее.
  • Закрытое состояние в виде кнопки внизу справа или слева.
  • Английский, русский, украинский и польский интерфейсы.
  • Автоматическое чтение языка браузера и английский fallback.
  • Светлая, тёмная и автоматическая темы.
  • Настройка цветов, текстов, позиции и закрытой иконки.
  • Передача описания проекта, услуг, цен, FAQ, страниц, контактов и правил.
  • Передача имени посетителя и текущей страницы.
  • Backend на Cloudflare Workers.
  • Последовательное переключение Groq → Gemini → Workers AI.
  • Сохранение последних сообщений при переключении провайдера.
  • События завершённого обмена сообщениями и закрытия беседы.
  • Публичная демонстрация на GitHub Pages.

2.2. Чего пока нет

  • Автоматического сканирования или индексирования сайта.
  • Постоянной памяти после перезагрузки страницы.
  • Серверной базы разговоров.
  • Готовой отправки email, Telegram или CRM-уведомлений.
  • Проверки личности пользователя.
  • Отдельных лимитов на каждого посетителя.
  • Готового личного кабинета, аналитики и биллинга.
  • Бронирований, оплат и автономных действий.
  • RAG/векторной базы для большого объёма документов.
  • Гарантии абсолютной фактической точности ответа модели.
Важно: project-данные направляют модель, но не «обучают» её навсегда. Данные повторно передаются вместе с каждым запросом и действуют только как контекст текущего вызова.

5. Полный жизненный цикл сообщения

  1. Браузер загружает demo-страницу и React bundle.
  2. App получает public URL Worker из VITE_ASSISTANT_API_URL.
  3. App определяет язык и передаёт его в AiAssistant.
  4. Компонент создаёт sessionId через crypto.randomUUID(), а при отсутствии — через время и случайное число.
  5. В состоянии появляется локальное приветственное сообщение. Оно ещё не получено от AI.
  6. Посетитель открывает виджет. Компонент ставит focus в input.
  7. Посетитель вводит до 500 символов и отправляет форму.
  8. useAssistant обрезает пробелы, добавляет user message и берёт последние maxHistory сообщений.
  9. Если передан transport, используется он. Иначе из apiUrl создаётся стандартный transport.
  10. Стандартный transport выполняет POST {apiUrl}/chat с JSON.
  11. Worker проверяет Origin, метод, path, общий размер body и JSON.
  12. Worker проверяет siteId, sessionId, locale, messages, project, visitorName и page.
  13. Worker строит system prompt. В него входят project, язык, имя и страница.
  14. Router читает AI_ROUTES слева направо.
  15. Провайдер без секрета пропускается. Провайдер в cooldown пропускается временно.
  16. Первый успешный провайдер возвращает текст.
  17. Worker возвращает message и диагностический meta.
  18. Стандартный transport берёт только message.content и отдаёт его hook.
  19. Hook добавляет assistant message в UI.
  20. Создаётся событие assistant.exchange.completed.
  21. Событие независимо передаётся в onEvent и, если настроено, в eventDelivery.endpoint.
Если посетитель отправляет следующее сообщение, project и последние сообщения отправляются снова. Backend не достаёт историю из БД: источником истории сейчас является состояние React-компонента.

6. Автоматическое определение языка

6.1. Поддерживаемые значения

КодЯзыкПримеры browser tags
enEnglishen, en-US, en-GB
ruРусскийru, ru-RU
ukУкраїнськаuk, uk-UA, uk_UA
plPolskipl, pl-PL

6.2. Порядок выбора в demo

  1. Читается localStorage["first-ai-locale:demo"].
  2. Если значение является одним из четырёх поддерживаемых — оно побеждает. Это сохраняет ручной выбор посетителя.
  3. Если сохранённого значения нет, вызывается detectBrowserLocale().
  4. Функция читает navigator.languages. Это упорядоченный браузером список предпочтений.
  5. Каждый tag приводится к нижнему регистру, _ заменяется на -, региональная часть отбрасывается.
  6. Выбирается первый поддерживаемый язык.
  7. Если браузерного API нет, чтение вызвало ошибку, список пуст или языки не поддерживаются — используется en.
Fallback гарантирован: английский хранится в DEFAULT_LOCALE и возвращается для неизвестных языков, например de-DE, fr-FR или es-ES.

Это не HTTP-запрос и не запрос разрешения пользователя. navigator.languages — стандартное значение, уже предоставленное браузером. Геолокация, IP и cookies для определения языка не используются.

6.3. Поведение при ручном переключении

  • Верхний переключатель demo меняет язык страницы.
  • Selector внутри виджета вызывает onLocaleChange, поэтому demo и чат синхронизируются.
  • Выбранный код записывается в localStorage для текущего siteId.
  • При изменении языка input очищается, текущий запрос отменяется, создаётся новый sessionId и история сбрасывается к приветствию.
  • Сброс нужен, чтобы system instruction нового языка не смешивалась с предыдущей беседой.

6.4. Использование в другом проекте

<AiAssistant

  project={project}

8. Подключение компонента к другому React-проекту

8.1. Установка

npm install @first-ai/assistant-react

Пока пакет не опубликован в принадлежащем владельцу npm scope, используется локальный tarball:

# В репозитории First AI
npm run build:widget
npm pack --workspace @first-ai/assistant-react

# В другом проекте
npm install /absolute/path/to/first-ai-assistant-react-0.1.0.tgz

8.2. Минимальное подключение

import { AiAssistant } from "@first-ai/assistant-react";
import "@first-ai/assistant-react/styles.css";
import projectJson from "./assistantProject.json";
import type { AssistantProjectData } from "@first-ai/assistant-react";

const project = projectJson as AssistantProjectData;

export function App() {
  return (
    <AiAssistant
      siteId={project.id}
      project={project}
      apiUrl="https://your-worker.workers.dev"
      locale="auto"
    />
  );
}

Обязательно подключить CSS. В apiUrl указывается base URL без /chat: библиотека добавит путь сама.

8.3. Что должен подготовить новый проект

  1. Публичный assistantProject.json.
  2. Уникальный siteId, совпадающий с project.id.
  3. Worker URL или собственный transport.
  4. Разрешённый Origin в конфигурации Worker.
  5. При необходимости — backend endpoint для events.
  6. Решение, откуда брать visitorName: auth state сайта или не передавать.
  7. Политику согласия, хранения и обработки беседы.

9. Все props компонента AiAssistant

Ниже описан каждый prop из AiAssistantProps. «Передаётся в chat request» означает, что значение уходит на backend. «Только UI» означает, что Worker это значение не получает.

PropТип / defaultПодробное поведение
siteIdstring, обязательныйПубличный ID установки. Уходит в chat request и events. Допустимый backend-формат: ^[a-z0-9-]{1,64}$. При наличии project обязан совпадать с project.id. Также входит в localStorage keys языка и темы. Это не user ID и не секрет.
apiUrlstring?Base URL backend без /chat. Последний slash удаляется. Если указан https://api.example.com/, запрос уйдёт на https://api.example.com/chat. Не содержит provider key.
transportAssistantTransport?Собственная async-функция. Имеет приоритет над apiUrl. Получает готовый AssistantRequest и AbortSignal, возвращает строку ответа. Полезна для proxy, тестов, другого API-контракта или local demo. Если нет ни transport, ни apiUrl, отправка завершится ошибкой.
projectAssistantProjectData?Публичная база знаний установки. Уходит с каждым запросом. Для универсальной работы на разных сайтах рекомендуется передавать всегда. Если отсутствует, Worker сможет ответить только для siteId, заранее записанного в sites.ts.
visitorNamestring?Имя, которое уже знает host-сайт. Обрезаются внешние пробелы. Показывается вместо «Вы», используется для двухбуквенного avatar, отправляется в request, prompt и events. Backend принимает до 100 символов. Не подтверждает личность.
pageContextPartial<AssistantPageContext>?Переопределяет автоматически найденные url, path, title. Для каждого отсутствующего поля берётся значение браузера. Page отправляется только если в итоге есть все три поля.
eventDeliveryAssistantEventDelivery?Настраивает отправку событий на backend. Сейчас поддерживается объект { type: "backend", endpoint, events? }. Не влияет на chat response. Ошибка events endpoint не удаляет уже показанный AI-ответ.
titlestring?Только UI. Заголовок в header чата. Если отсутствует, берётся из словаря активного языка.
eyebrowstring?Только UI. Маленькая строка над title, например «Онлайн-помощник».
statusTextstring?Только UI. Текст рядом с зелёной status-точкой. Не является реальной проверкой состояния provider; это статическая надпись.
welcomeMessagestring?Только начальное сообщение интерфейса. Создаётся локально, не является AI-ответом. При смене языка или reset история возвращается к этому сообщению.
placeholderstring?Только UI. Placeholder input.

15. Формирование system prompt

15.1. С project

Worker создаёт инструкцию, которая:

  • называет проект;
  • задаёт язык ответа;
  • разрешает использовать только PROJECT_DATA как источник бизнес-фактов;
  • запрещает придумывать цены, скидки, сроки, URLs и возможности;
  • отдельно применяет assistantRules;
  • разрешает советовать только URL из project;
  • просит собирать контакт, но не утверждать, что уведомление доставлено;
  • включает runtime context с visitorName и page;
  • включает сериализованный project JSON.

Одинаковая system instruction передаётся каждому provider route. Поэтому при fallback новая модель видит те же факты и последние messages.

15.2. Без project

Worker ищет siteId в apps/api/src/sites.ts. Сейчас встроен только demo с четырьмя короткими language prompts. Для универсальной интеграции project следует передавать.

16. Маршрутизация Groq, Gemini и Workers AI

16.1. Production order

1. groq:openai/gpt-oss-120b
2. groq:openai/gpt-oss-20b
3. gemini:gemini-3.5-flash
4. workers-ai:@cf/zai-org/glm-4.7-flash

16.2. Правила fallback

  1. Если adapter не настроен, route = skipped.
  2. Если route имеет cooldown, он не вызывается.
  3. Каждый вызов ограничен PROVIDER_TIMEOUT_MS, production = 20000 ms.
  4. При успехе cooldown удаляется и возвращается результат.
  5. При 429 используется Retry-After или 60 секунд cooldown.
  6. Другие retryable-ошибки получают 10 секунд cooldown.
  7. Authentication/configuration не останавливают цепочку: router пробует следующий route.
  8. Неретрайная invalid request ошибка не маскируется fallback.
  9. Если успеха нет — AllProvidersUnavailableError и внешний 502.

Cooldown хранится в памяти конкретного Worker isolate, поэтому это оптимизация, а не глобальная долговечная таблица состояния.

16.3. Groq adapter

  • Secret: GROQ_API_KEY.
  • Endpoint: https://api.groq.com/openai/v1/chat/completions.
  • Authorization Bearer добавляется только Worker.
  • temperature = 0.3, max_completion_tokens = 500.

16.4. Gemini adapter

  • Secret: GEMINI_API_KEY.
  • Endpoint: Google Generative Language v1beta generateContent.
  • Ключ передаётся server-side header x-goog-api-key.
  • Assistant messages преобразуются в role=model.
  • temperature = 0.3, maxOutputTokens = 500.

16.5. Workers AI adapter

  • Не требует Groq/Gemini key.
  • Использует Cloudflare binding env.AI.
  • Вызов AI.run(model, payload) выполняется внутри Worker.
  • temperature = 0.3, max_tokens = 500.

16.6. Лимиты

Лимит provider относится к аккаунту/проекту, чьи credentials установлены в Worker. Два посетителя не получают отдельные «1000 запросов»; они расходуют общий пул. siteId не создаёт отдельный billing account и не разделяет quota.

17. События беседы и передача обратной связи

AI-ответ возвращается прямо в компонент через /chat. Отдельно component может сообщить владельцу сайта, о чём шёл разговор. Для этого используются events.

17.1. Рекомендуемая production-схема

AiAssistant
   │ POST /api/assistant-events
   ▼
Backend владельца сайта
   ├─ проверка schema
   ├─ consent / privacy
   ├─ сохранение в БД
   ├─ выделение контакта/lead
   ├─ email менеджеру
   ├─ Telegram Bot API
   └─ CRM webhook

В eventDelivery.endpoint можно передать относительный URL, например /api/assistant-events, чтобы запрос ушёл на тот же домен host-сайта. Секреты email/Telegram/CRM остаются на этом backend.

17.2. Events filter

eventDelivery={{
  type: "backend",
  endpoint: "/api/assistant-events",
  events: ["assistant.conversation.closed"]
}}

Если events отсутствует, отправляются оба типа. Если массив указан, остальные события отфильтровываются.

17.3. Транспорт events

  • method POST;
  • Content-Type application/json;
  • keepalive: true, чтобы повысить шанс доставки при уходе со страницы;
  • delivery выполняется асинхронно и не блокирует UI;
  • неуспешный HTTP status превращается в Error и передаётся callback.

21. Безопасность и публичность данных

21.1. Уже реализовано

  • Exact Origin allowlist.
  • Строгие paths и methods.
  • Ограничение request body.
  • Проверка IDs, locale, roles, message content, URLs и project arrays.
  • Project нормализуется и только разрешённые fields попадают в prompt.
  • Provider errors наружу заменяются общими сообщениями.
  • No-store responses.
  • Secrets только server-side.

21.2. Что необходимо перед публичным коммерческим запуском

  • Rate limiting на Worker.
  • Turnstile или другой anti-abuse механизм.
  • Ограничения по site/customer и отдельные quotas.
  • Authentication между host backend и event endpoint.
  • Consent перед сохранением разговоров.
  • Retention policy и удаление персональных данных.
  • Monitoring стоимости и provider quota.
  • Разделение tenants и project data на server side при SaaS-модели.
  • Server-side schema validation events в backend владельца.
CORS защищает обычные browser calls, но не мешает злоумышленнику отправить прямой scripted HTTP request с поддельными данными. Поэтому CORS не заменяет rate limit и authentication.

23.4. Собственный API contract

const transport: AssistantTransport = async (request, signal) => {
  const response = await fetch("/api/my-assistant", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ assistantRequest: request }),
    signal
  });
  const data = await response.json();
  return data.answer;
};

<AiAssistant siteId={project.id} project={project} transport={transport} />

24. Ограничения и рекомендуемое развитие

24.1. Ближайшие технические улучшения

  1. Rate limiting + Turnstile.
  2. Настоящий event receiver с DB и notifications.
  3. Server-side project registry: frontend передаёт projectId, а private/large knowledge хранится backend.
  4. Persistent conversations и resume token.
  5. Tenant/customer keys и отдельные quotas.
  6. RAG для документов и страниц.
  7. Структурированный lead extraction.
  8. Admin dashboard, usage и costs.
  9. Tool calling для bookings/CRM при строгих разрешениях.
  10. Версионирование API и schema migrations.

24.2. Когда props-first project уже недостаточно

Если данных больше 15 KB, они приватные, часто меняются или принадлежат многим tenants, project не следует целиком отправлять из браузера. Тогда frontend передаёт стабильный project/customer ID, Worker аутентифицирует установку и получает знания из KV/D1/R2/Vectorize или внешней БД.

27. Короткая модель понимания

Что передаём компоненту: публичное описание проекта, имя текущего пользователя при наличии, язык, UI-настройки, public backend URL и event endpoint.

Что компонент отправляет: site/session IDs, конкретный locale, последние сообщения, project, optional visitor/page.

Что делает Worker: валидирует, создаёт prompt, добавляет server secret, вызывает модели по очереди и возвращает ответ.

Что возвращается: assistant message и metadata о provider attempts.

Что получает владелец сайта: structured events, если настроил свой endpoint.

Что нигде нельзя показывать: AI keys, bot tokens, DB/email credentials.

Документ подготовлен по фактическим исходникам First AI v0.1: React package, demo app, Cloudflare Worker, runtime validators, provider adapters, tests, JSON Schema и GitHub Pages workflow.
